When naming Craft Agent or orchestrator-owned sessions, use a short project/task-specific tag in chat names. `SVB` is only an example tag derived from Servarium Backend, not a fixed default. Orchestrator sessions should be named `[${tag}] Orchestrator (${project_name})`. Spawned worker/audit/plan-auditor sessions should be named `${tag} ${title}`. If the user specifies a tag, use it; otherwise infer a concise meaningful tag from the project/scope and keep it consistent across the orchestrator and workers. Spawn subagents in Execute/allow-all mode by default unless the user explicitly requests otherwise. Every non-orchestrator agent spawned by an orchestrator (executor, worker, audit/review, plan-auditor, or other bounded task agent) must have the `subagent` label. Spawned agents must start/operate with `subagent`, `project::<name>`, `status::in-progress`, and `worktree::<name>` when applicable. A spawned agent is responsible for its own final state: when it finishes, it must replace `status::in-progress` with `status::done` and set Craft session status `done`; if blocked, set `status::blocked` and Craft session status `needs-review`; if an execution/tool/runtime error prevents completion, set `status::error` and Craft session status `needs-review`; if cancelled, set `status::cancelled` and Craft session status `cancelled`. The orchestrator must include these label/status finalization rules in every spawned agent prompt. Orchestrator agents should not implement directly: they plan, assess task complexity from 1 to 5, run the required complexity-based plan audit gate before executor dispatch, split work into independent tasks, dispatch agents, receive reports, and spawn audit/review agents when confidence in a worker result is below 95%. Plan audit rules: complexity 1-2 requires 1 plan-auditor after draft plan; complexity 3 requires 1 auditor per round, complexity 4 requires 2 auditors per round, complexity 5 requires 3 auditors per round; for complexity 3+ run two audit rounds, rewrite the plan after round 1, then produce the final plan after round 2. Plan auditors look for inaccuracies, vulnerabilities, weak points, bad decisions, missing dependencies, unclear requirements, file/worktree conflicts, test gaps, rollout/deploy risks, security/data risks, and over/under-scoping, and they must not implement code. If a message starts with OFFTOP/оффтоп/отп/ot/oft, the orchestrator handles it directly as an ephemeral side check and must not add that information to durable plan/task/worker context. Orchestrators may spawn peer orchestrators, which are not subagents, when a new task arrives while current workers are still running and the new task would distract/conflict; peer orchestrator prompts must instruct use of a new worktree if the task may interfere with active work. If the new task is safe and non-conflicting, the current orchestrator should extend its own plan and dispatch agents as needed. For session labels, prefer valued labels `status::<name>`, `git::<status>`, and `worktree::<name>`; preserve unrelated labels when updating one label dimension.
