When naming Craft Agent or orchestrator-owned sessions, use a short project/task-specific tag. `SVB` is only an example, not a fixed default. Orchestrator sessions use `[${tag}] Orchestrator (${project_name})`; spawned worker sessions use `${tag} ${title}`.

`craft-agent-workflow/SKILL.md` is canonical for shared workflow rules. Every session coordinating other sessions must self-apply and preserve `orchestrator`, especially before changing another session's Craft status. Every non-orchestrator agent must have `subagent`, `project::<name>`, `status::in-progress`, `worktree::<name>` when applicable, and exactly one mandatory primary role label: `executor` for implementation, `auditor` for audit/review and plan audit, `designer` for product/UX/UI design, or `tester` for QA/testing/verification. Role workers read current labels, remove conflicting primary-role labels, preserve unrelated project/worktree/git/priority and appropriate status labels, then add their own role.

Approved scope is the explicit objective, acceptance criteria, named boundaries, and directly necessary changes; ordinary in-scope implementation choices do not need repeated permission. Never change anything outside that scope without explicit user approval. Out-of-scope findings remain report-only regardless of `P0`, security, or data-loss severity. Orchestrators must obtain user approval before adding them to plans or prompts; child agents stop and report instead of self-expanding. Any role may take only the actions strictly necessary to stop an actively harmful operation and preserve state/evidence, but may not repair it unless the repair is already in approved scope or the user explicitly approves scope expansion.

Load the matching canonical role skill. Plan auditors load both `craft-agent-auditor` and `plan-auditor`; the plan-auditor report schema supersedes the generic auditor schema. The generic `Result:` report is executor-only. Every role report includes exact `- Confidence: <0–100>%` and `- Confidence rationale:` lines.

Audit, design, test, and plan-auditor workers do not implement, commit, or push. They report implementation needs; a separate executor session loading `craft-agent-executor` and labeled `executor` may be created only when the work is already within approved scope or after the user explicitly approves scope expansion. The automatic below-85% result-audit trigger applies only to executor, designer, and tester results. A low-confidence auditor or plan-auditor result is terminally handled by resolving evidence, one bounded discovery/retest task, documented risk acceptance/deferral, or blocked status; it must not start an unbounded audit chain.

Spawn executor agents in Execute/allow-all by default when supported; audit/review agents may use safe/read-only modes. Spawned agents own their final state: preserve unrelated labels, replace `status::in-progress` with `status::done`, `status::blocked`, `status::error`, or `status::cancelled`, and set Craft session status to `needs-review` by default. Executor auto-close is allowed only when the prompt contains exactly `auto-close on success: true` and all verification passes; it never applies to audit/review agents or non-success outcomes. Orchestrators plan rather than implement, assess complexity 1–5, run the mandatory plan-audit gate, dispatch independent work in parallel, and collect reports. Complexity 1–2 requires one plan auditor; complexity 3 requires one auditor per round, complexity 4 two, complexity 5 three; complexity 3+ requires two rounds. Treat OFFTOP/оффтоп/отп/ot/oft messages as ephemeral side checks. Peer orchestrators are not subagents. Preserve unrelated labels whenever updating `status::<name>`, `git::<status>`, or `worktree::<name>`.
